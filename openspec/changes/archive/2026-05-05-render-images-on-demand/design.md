## Context

当前系统在 `server/rss/worker.go` 的抓取阶段完成完整图片管线：提取图片 URL、下载图片、缩放到 320x240、叠加文字、保存到 `data/images/...`，并将本地路径存入 `items.image_path`。`server/image/handler.go` 仅根据 item ID 查出本地路径后调用 `http.ServeFile` 返回静态 JPEG。

这次变更会跨越抓取、数据模型、图片接口、配置和指标几个模块，但不引入新的外部服务，也不要求兼容旧数据。项目约束仍然不变：Go 1.24、纯 Go 依赖、无迁移文件、通过 `AutoMigrate` 管理 schema、运行时配置来自 `config.yaml`。

## Goals / Non-Goals

**Goals:**
- 将 RSS 抓取阶段改为只保存条目元数据和原始图片 URL
- 让 `/image/{id}.jpg` 成为每个条目的统一图片入口，而不是只服务已有本地文件
- 在访问图片接口时，以配置化超时下载上游图片，成功时缩放并叠加标题/发布时间，失败时回退为纯色色卡
- 保留现有的图片提取策略、色卡样式和文字叠加效果
- 为按需渲染补充清晰的日志和 Prometheus 指标

**Non-Goals:**
- 不实现图片缓存、磁盘缓存或内存缓存
- 不兼容历史 `image_path` 数据，也不做数据迁移脚本
- 不优化设备端等待时间或客户端重试策略
- 不改变 `/v1/device/{device_id}/next` 的响应结构

## Decisions

### 1. 以原始图片 URL 替代本地图片路径持久化

**决策**：将 `models.Item` 中的图片字段改为保存提取到的原始图片 URL，而不是本地 JPEG 文件路径。

**理由**：新架构的核心是“抓取时不做图片处理”，因此数据库里需要保存的是图片来源，而不是处理产物。这样抓取阶段保持轻量，图片接口可按条目独立渲染。

**替代方案**：同时保留 `image_path` 和 `image_url` 双字段，或以兼容方式过渡。因为用户明确接受删除旧数据并重新开始，这种兼容层只会增加复杂度。

### 2. `/image/{id}.jpg` 统一服务所有条目

**决策**：`/v1/device/{device_id}/next` 对每个 item 都返回本地 `/image/{id}.jpg`，不再根据条目是否提取到原始图片 URL 决定是否返回 `image_url`。

**理由**：图片接口现在承担两种职责：渲染真实图片，或在无图/失败时生成纯色色卡。如果仍然只对“有原图 URL”的条目返回 `image_url`，则无图条目无法走统一渲染路径，和本次目标冲突。

**替代方案**：仅当存在原始图片 URL 时才返回 `image_url`。这会保留旧语义，但会让“无图条目也应返回色卡图片”的目标无法达成。

### 3. 图片接口现场完成完整渲染

**决策**：把下载、超时控制、解码、缩放、文字叠加、JPEG 编码全部移动到 `server/image/handler.go`，handler 直接写响应，不写磁盘。

**理由**：这是最直接的职责重构。图片接口本来就是最终图片消费者的入口，将渲染逻辑集中到这里，可以让抓取流程和服务流程边界更清晰。

**替代方案**：抽出新的共享服务层，由 worker 和 handler 共同调用。当前只有 handler 需要渲染，先集中在图片域内更简单；若未来出现缓存或后台预热，再考虑进一步抽象。

### 4. 下载失败统一回退为纯色色卡

**决策**：当条目没有原始图片 URL、下载超时、HTTP 非 200、解码失败或格式不支持时，图片接口统一生成纯色色卡，并叠加标题与 RSS 发布时间。

**理由**：这使图片接口具备稳定输出，不把上游图片失败扩散为 404 或 5xx。用户已经明确接受“失败即当作没有照片处理”。

**替代方案**：返回 404、502 或空响应。这样会把上游不稳定性直接暴露给设备端，不符合这次探索收敛出的降级策略。

### 5. 下载超时通过配置控制，初始值为 3 秒

**决策**：在 `rss` 配置下新增图片下载超时秒数字段，由图片 handler 读取；默认和首个使用值为 3 秒。

**理由**：超时是运行时策略，不应硬编码。放在现有 `rss` 配置区能保持图片采集与渲染策略集中。

**替代方案**：写死在代码里，或放到 `server` 配置下。写死不利于后续调优；放在 `server` 下会弱化它与 RSS 图片流程的关联。

### 6. 保留现有图片提取与排版逻辑

**决策**：继续复用当前 worker 中成熟的 7 层图片 URL 提取逻辑、字体与色卡逻辑，只调整调用位置与生命周期。

**理由**：本次变化是处理时机重构，而不是视觉或提取算法重写。复用现有逻辑能减少行为漂移。

**替代方案**：同时重写图片提取策略或重新设计视觉样式。这样会把一个明确的架构切换扩张成多目标改造，增加风险。

## Risks / Trade-offs

- [上游图片变慢会直接拉长图片请求时延] → 用户明确接受慢响应，不额外引入缓存；通过 3 秒超时上限控制最差路径
- [相同图片被重复请求时会重复下载和渲染] → 这是刻意选择的简单性取舍，本次不实现缓存
- [图片 handler 变重，职责比现在复杂] → 将图片相关逻辑集中在单一入口，避免 worker 与 handler 各保留半套流程
- [数据库字段语义变化会导致旧数据失效] → 用户接受清空旧数据重新开始，不做兼容层
- [观测点从抓取阶段迁移到服务阶段后，错误分布会变化] → 新增渲染与回退指标，并保留关键失败日志，便于重新建立运行时认知

## Migration Plan

1. 更新数据模型与配置结构
2. 删除旧数据库和本地图片数据目录
3. 启动服务，通过 `AutoMigrate` 生成新 schema
4. 重新抓取 RSS 条目，写入新的图片 URL 数据
5. 验证 `/next` 与 `/image/{id}.jpg` 的新行为

回滚方式：回滚代码并重新初始化数据库。由于本次不保留兼容字段，回滚同样依赖重建数据。
